# VAT & EPR 动态表单系统 - 技术方案介绍

> 面向团队的技术方案说明文档

---

## 📋 一、这个项目是做什么的？

### 业务背景
我们为跨境电商企业提供 **VAT（增值税）** 和 **EPR（环境产品注册）** 的申报服务。不同国家（德国、法国、意大利等）的申报要求不同，每个业务场景需要的表单字段也不一样。

### 核心痛点
- ❌ 每个新业务都要开发新表单，重复工作量大
- ❌ 字段调整需要改代码、重新发版
- ❌ 不同国家的表单差异难以统一管理

### 我们的解决方案
**"搭积木"式的动态表单系统**：
- ✅ 管理员通过拖拽设计表单模板，无需写代码
- ✅ 支持 7 种控件类型（输入框、下拉框、日期、上传等）
- ✅ 可视化配置业务流程（提交→审核→递交→完成）
- ✅ 一个系统适配所有国家和业务类型

## 🏗️ 二、系统架构（三层抽象模型）

```
┌─────────────────────────────────────────────────────────┐
│                    业务人员使用流程                       │
│                                                         │
│  第1层：控件管理  →  第2层：模板设计  →  第3层：服务单填写  │
│  (定义积木块)        (搭建表单)          (使用表单)         │
└─────────────────────────────────────────────────────────┘
```

### 1️⃣ 控件管理 - "定义积木块"
**目标**：定义表单中可以使用的字段类型

**示例控件**：
| 控件类型 | 用途 | 示例 |
|---------|------|------|
| INPUT | 单行文本 | 公司名称 |
| SELECT | 下拉选择 | 国家代码 |
| DATE | 日期选择 | 注册日期 |
| UPLOAD | 文件上传 | 营业执照 |

**关键设计 - controlKey**：
每个控件都有一个唯一的 `controlKey`，格式为 `类名.字段名`
```
Company.companyName         → 映射到 Company 实体的 companyName 字段
CompanyLegalPerson.legalName → 映射到 CompanyLegalPerson 实体的 legalName 字段
```

### 2️⃣ 模板管理 - "搭建表单"
**目标**：用控件拼装出完整的业务表单

**三步配置流程**：
```
步骤1：基础信息          步骤2：表单配置          步骤3：流程配置
- 模板名称              - 拖拽控件到画布          - 可视化连线
- 服务类型(VAT/EPR)     - 调整布局(1-4列)        - 配置流转规则
- 国家代码              - 设置校验规则            - 预设VAT/EPR流程
- 服务类目(三级联动)    - 保存为JSON Schema       - 一键加载
```

**模板示例**：
```
德国VAT注册模板
├── 公司信息（2列布局）
│   ├── Company.companyName    [输入框]
│   ├── Company.taxNumber      [输入框]
│   └── Company.registrationDate [日期]
├── 法人信息（2列布局）
│   ├── CompanyLegalPerson.legalName [输入框]
│   └── CompanyLegalPerson.idCard    [上传]
└── 流程配置
    └── 待提交 → 待审核 → 待递交 → 当地同事处理 → 完成
```

### 3️⃣ 服务单实例 - "使用表单"
**目标**：基于模板创建实际的业务单据

**工作流程**：
```
选择模板 → 创建服务单 → 填写表单 → 保存草稿 → 提交审核 → 流程流转
```

**业务状态机**：
```
10 待提交 ──submit──→ 20 待审核
20 待审核 ──auditPass──→ 30 待递交
20 待审核 ──auditReject──→ 50 已驳回
30 待递交 ──submitLocal──→ 33 当地同事处理
33 当地同事处理 ──submitOrg──→ 31 组织处理
31 组织处理 ──complete──→ 40 已完成
50 已驳回 ──resubmit──→ 20 待审核
```

## 💡 三、核心技术亮点

### 亮点1：动态表单渲染引擎
**问题**：如何根据 JSON 配置渲染出完整表单？

**解决方案**：
```javascript
// 前端收到的 JSON Schema
{
  "controls": [
    {
      "controlKey": "Company.companyName",
      "label": "公司名称",
      "type": "INPUT",
      "required": true,
      "span": 2  // 占2列
    }
  ]
}

// 前端动态渲染
<component :is="getComponent(control.type)" v-bind="control" />
```

**优势**：新增控件类型只需注册一次，所有模板自动支持

---

### 亮点2：FormDataConverter - 智能数据映射
**问题**：表单提交的数据是扁平的 Map，如何转换为业务实体？

**转换过程**：
```
前端提交的数据：
{
  "Company.companyName": "测试公司",
  "Company.taxNumber": "DE123456789",
  "CompanyLegalPerson.legalName": "张三"
}
         ↓  FormDataConverter 处理
后端得到的实体：
{
  Company: {
    companyName: "测试公司",
    taxNumber: "DE123456789"
  },
  CompanyLegalPerson: {
    legalName: "张三"
  }
}
         ↓  反射创建对象
Java 对象：
new Company().setCompanyName("测试公司").setTaxNumber("DE123456789")
new CompanyLegalPerson().setLegalName("张三")
```

**实现原理**：
1. 按 `controlKey` 中的类名分组
2. 反射创建对应实体对象
3. 调用 setter 方法赋值
4. 返回完整的业务对象树

---

### 亮点3：可视化工作流配置
**问题**：如何让非技术人员也能配置业务流程？

**解决方案**：使用 Vue Flow 实现拖拽式流程图

**操作方式**：
```
1. 左侧选择状态节点（待提交、待审核等）
2. 拖拽到画布中
3. 从一个节点底部拖拽连线到另一个节点
4. 在弹出的面板中配置：
   - 动作名称（提交、审核通过、驳回等）
   - 是否需要备注
   - 适用条件（通用/VAT/EPR）
5. 保存后自动生成 JSON 配置
```

**预设流程**：
- 🟢 **默认流程**：适合简单业务
- 🔵 **VAT流程**：待提交→待审核→待递交→当地同事处理→税局处理→完成
- 🟡 **EPR流程**：待提交→待审核→待递交→当地同事处理→组织处理→完成

## 🔍 八、关键技术点详解

### 1. JSON Schema 设计
模板的核心是 JSON Schema，描述了表单的完整结构：

```json
{
  "templateName": "德国VAT注册",
  "serviceType": "VAT",
  "countryCode": "DEU",
  "schema": {
    "controls": [
      {
        "controlKey": "Company.companyName",
        "label": "公司名称",
        "type": "INPUT",
        "required": true,
        "span": 2,
        "rules": {
          "minLength": 2,
          "maxLength": 100
        }
      }
    ]
  }
}
```

### 2. 三级联动服务类目
```
一级：VAT / EPR
  ├─ 二级（VAT）：注册 / 申报 / 注销
  │   └─ 三级：德国VAT注册、法国VAT申报...
  │
  └─ 二级（EPR）：包装法 / WEEE法 / 电池法
      └─ 三级：德国包装法、法国WEEE法...
```

### 3. 工作流配置 JSON
```json
{
  "transitions": [
    {
      "from": 10,
      "to": 20,
      "action": "submit",
      "actionName": "提交",
      "needRemark": false,
      "condition": null
    },
    {
      "from": 20,
      "to": 50,
      "action": "auditReject",
      "actionName": "审核驳回",
      "needRemark": true,
      "condition": null
    }
  ],
  "allowTerminateFrom": [10, 20, 30, 50]
}
```

## 💬 十一、常见问题

### Q1：为什么要用 controlKey 而不是直接用字段名？
**A**：controlKey 包含了类名信息（如 `Company.companyName`），这样后端才知道要把值赋给哪个实体对象。如果只用 `companyName`，多个实体都有这个字段时会冲突。

### Q2：JSON Schema 存在哪里？
**A**：存在 `form_template` 表的 `schema` 字段中，是 JSON 格式的字符串。前端读取后动态渲染表单。

### Q3：工作流配置是硬编码的吗？
**A**：不是。每个模板可以配置自己的工作流，存在 `workflow_transition` 表中。系统提供了 VAT 和 EPR 的预设配置，也可以自定义。

### Q4：如何支持新的控件类型？
**A**：
1. 后端：在 `enums/ControlType.java` 添加新类型
2. 前端：在 `components/DynamicForm/` 目录下创建新组件
3. 在设计器的控件面板注册新类型

### Q5：表单数据如何保证安全性？
**A**：
- 前端：根据控件配置的规则进行校验
- 后端：提交时再次校验必填项、长度等
- controlKey 映射时只处理白名单中的实体类
